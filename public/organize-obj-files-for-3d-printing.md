## An OBJ organization workflow

1. **Keep OBJ files in the same archive**: Avoid letting OBJ assets drift into separate tool-specific silos when they belong to the same printable model lineage.
2. **Tag by role**: Use tags that explain whether the OBJ is a source asset, printable derivative, support element, or project component.
3. **Version the important changes**: Keep revisions attached to the model so later retrieval does not depend on guessing which export is current.

## Where OBJ organization usually breaks

- Source files separated from printable variants.
- Project folders that do not reflect the final print workflow.
- Weak naming conventions for derivatives and revisions.
- No consistent place to preview the model before use.

## What to tag on OBJ assets

- Role: source, printable, support, remix, accessory.
- Project or client context.
- Status and approval level.
- Related kit or part-family labels.

## FAQ

### Can STL Shelf organize OBJ and STL files together?

Yes. Mixed-format libraries are part of the product positioning.

### Why should OBJ files live in the same library as STL files?

Because they often belong to the same model lineage or project context. Splitting them across tools makes retrieval weaker.

### Can STL Shelf preview OBJ files in browser?

Browser preview is part of the product workflow and helps reduce retrieval mistakes before printing.

### Is STL Shelf a public repository for OBJ files?

No. It is a private library software product, not a public sharing platform.
